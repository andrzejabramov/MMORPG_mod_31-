# Техническое задание: Фанатский портал MMORPG

## 1. Общая информация

### 1.1. Название проекта
**MMORPG Fan Board** — Доска объявлений для фанатского сервера MMORPG.

### 1.2. Цель проекта
Разработка интернет-ресурса, представляющего собой доску объявлений для фанатского сервера MMORPG. Платформа должна обеспечивать регистрацию пользователей, создание и управление объявлениями, систему откликов и новостные рассылки.

### 1.3. Технологический стек
- **Backend:** Python 3.10+, Django 4.2+
- **База данных:** PostgreSQL (рекомендуется) / SQLite (для разработки)
- **Frontend:** HTML5, CSS3 (Bootstrap 5), JavaScript (базовый)
- **Дополнительные библиотеки:**
  - `django-summernote` или `django-ckeditor` — для WYSIWYG-редактора
  - `django-crispy-forms` — для стилизации форм
  - `Pillow` — для работы с изображениями
  - `python-dotenv` — для управления переменными окружения

## 2. Функциональные требования

### 2.1. Пользовательская система

#### 2.1.1. Регистрация
- Регистрация по email.
- После отправки формы регистрации пользователь получает на указанный email письмо с 6-значным кодом подтверждения.
- Без подтверждения кода вход в систему невозможен.
- Срок действия кода — 24 часа.

#### 2.1.2. Аутентификация
- Вход по email и паролю.
- Выход из системы (logout).

#### 2.1.3. Профиль пользователя
- Просмотр своего профиля.
- Возможность редактировать личные данные (имя, email).

### 2.2. Объявления (Posts)

#### 2.2.1. Создание объявления
- Доступно только авторизованным пользователям.
- Поля объявления:
  - **Заголовок** (обязательное, строка до 255 символов)
  - **Текст объявления** (обязательное, WYSIWYG-редактор)
  - **Категория** (обязательное, выбор из предопределенного списка)
- В тексте объявления через WYSIWYG-редактор можно вставлять:
  - Изображения (загрузка с компьютера или по ссылке)
  - Видео (YouTube, Vimeo — встраивание через iframe/embed)
  - Форматированный текст (жирный, курсив, списки)

#### 2.2.2. Редактирование объявления
- Только автор объявления может редактировать свой пост.
- Доступно изменение всех полей.

#### 2.2.3. Удаление объявления
- Только автор объявления может удалить свой пост.
- При удалении объявления также удаляются все связанные с ним отклики (каскадное удаление).

#### 2.2.4. Просмотр объявлений
- Главная страница — список всех объявлений (пагинация, 10 постов на страницу).
- Страница детального просмотра объявления.
- Фильтрация по категориям на главной странице.

### 2.3. Категории объявлений

Строго фиксированный список категорий:
1. Танки (Tanks)
2. Хилы (Healers)
3. ДД (Damage Dealers)
4. Торговцы (Merchants)
5. Гилдмастеры (Guild Masters)
6. Квестгиверы (Quest Givers)
7. Кузнецы (Blacksmiths)
8. Кожевники (Leatherworkers)
9. Зельевары (Potion Makers)
10. Мастера заклинаний (Spell Masters)

### 2.4. Отклики (Responses)

#### 2.4.1. Создание отклика
- Доступно только авторизованным пользователям.
- Отклик представляет собой простой текст (до 1000 символов).
- Один пользователь может оставлять неограниченное количество откликов на одно объявление.

#### 2.4.2. Управление откликами (для автора объявления)
- Приватная страница "Мои отклики", доступная только автору объявлений.
- На странице отображаются все отклики на объявления текущего пользователя.
- **Фильтрация:**
  - По конкретному объявлению (выпадающий список)
  - По статусу (принят/не принят)
- **Действия с откликами:**
  - **Принять отклик** — меняет статус отклика на "принят".
  - **Удалить отклик** — полностью удаляет отклик из базы.

#### 2.4.3. Просмотр статуса отклика (для автора отклика)
- Пользователь может видеть статус своих откликов (принят/ожидает) на странице объявления.

### 2.5. Уведомления (Email-рассылки)

#### 2.5.1. Системные уведомления
- **При регистрации:** Письмо с кодом подтверждения.
- **При создании отклика:** Автору объявления приходит уведомление о новом отклике.
- **При принятии отклика:** Автору отклика приходит уведомление о том, что его отклик принят.

#### 2.5.2. Новостные рассылки
- Возможность отправки массовых email-рассылок всем подтвержденным пользователям.
- Реализация через кастомную команду Django (`manage.py send_newsletter`).
- Интерфейс для создания рассылок должен быть доступен через административную панель.

## 3. Модели данных

### 3.1. User (пользователь)
- Наследование от `AbstractUser`
- Поле `is_verified` (boolean) — подтвержден ли email

### 3.2. OneTimeCode (одноразовый код)
- `user` (ForeignKey to User)
- `code` (CharField, max_length=6)
- `created_at` (DateTimeField)

### 3.3. Category (категория)
- `name` (CharField with choices)

### 3.4. Post (объявление)
- `author` (ForeignKey to User)
- `category` (ForeignKey to Category)
- `title` (CharField, max_length=255)
- `content` (TextField, WYSIWYG)
- `created_at` (DateTimeField)
- `updated_at` (DateTimeField)

### 3.5. Response (отклик)
- `post` (ForeignKey to Post)
- `author` (ForeignKey to User)
- `text` (TextField, max_length=1000)
- `is_accepted` (BooleanField, default=False)
- `created_at` (DateTimeField)

## 11. Примеры использования

### Сценарий 1: Регистрация и создание объявления
1. Пользователь заходит на сайт, нажимает "Регистрация".
2. Вводит email и пароль, нажимает "Зарегистрироваться".
3. Получает на email код, вводит его на странице подтверждения.
4. Перенаправляется на главную, видит кнопку "Создать объявление".
5. Заполняет форму (заголовок, текст с картинкой, категория), нажимает "Опубликовать".
6. Объявление появляется на главной.

### Сценарий 2: Отклик и уведомление
1. Авторизованный пользователь заходит на объявление другого пользователя.
2. Оставляет текстовый отклик.
3. Автор объявления получает email: "Новый отклик на ваше объявление".
4. Автор заходит в раздел "Мои отклики", видит новый отклик, нажимает "Принять".
5. Пользователь, оставивший отклик, получает email: "Ваш отклик принят".

## 12. Заметки для разработчика

- При разработке использовать ветки Git (main, develop, feature/*).
- Все секретные данные (SECRET_KEY, пароли от БД, email) хранить в переменных окружения (.env).
- Для отправки email в разработке использовать консольный бэкенд (`EMAIL_BACKEND = 'django.core.mail.backends.console.EmailBackend'`).
- При деплое обязательно отключать DEBUG и настраивать безопасность.
